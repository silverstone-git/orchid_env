# Data Forge: Map-Reduce Orchestrator Evaluation Environment

Data Forge is a specialized **OpenEnv** puzzle game designed to benchmark Large Language Models on their ability to act as **Orchestrators** for complex, large-context data processing tasks.

Instead of directly solving a problem, the agent must design an execution pipeline. The environment operates as a turn-based debugging loop where one episode equals one task:
1.  **Partition** a massive dataset (10,000+ lines) into chunks.
2.  **Delegate** processing to multiple sub-agents, providing each with custom Python extraction logic.
3.  **Synthesize** (Reduce) the outputs from all agents into a final, structured answer.
4.  **Iterate**: If the pipeline crashes or returns the wrong answer, the agent receives tracebacks and partial credit scores, allowing it to try again until it runs out of attempts.

## 🏗 Architecture & Execution Flow

The environment leverages **Local Code Execution** (within the containerized OS) to provide fast and efficient processing of Map-Reduce pipelines.

> [!WARNING]
> **SECURITY ADVISORY**: This environment executes arbitrary Python code generated by the LLM. For security reasons, it **MUST ONLY** be run inside a Docker container. Running it directly on your host machine is highly discouraged as it grants the LLM execution access to your local filesystem.

### The Map-Reduce Pipeline
When an agent submits an `OrchestratorAction`:
1.  **Map Phase**: The environment concurrently executes sub-agents using a `ThreadPoolExecutor`. For each agent:
    -   It maps the target dataset path.
    -   It slices the dataset log according to `start_line` and `end_line`.
    -   It injects the slice into a variable called `chunk_data`.
    -   It executes the agent's `python_code` in an isolated local process (within the container).
2.  **Reduce Phase**:
    -   All strings printed/returned by sub-agents are collected into a list called `sub_outputs`.
    -   The agent's `synthesis_code` is executed in a fresh process, which has access to `sub_outputs`.
3.  **Synthesis**: The final output of the synthesis script is graded.
4.  **Grader Output**: The environment calculates multi-objective scores and returns detailed feedback, populating `execution_output`, `sub_agent_outputs`, and partial credit metrics in the `OrchestratorObservation` so the agent can debug its next attempt.

## 🏆 Multi-Objective Reward Function

The reward signal is designed to guide the agent toward "wise" orchestration, not just correct answers. It uses fine-grained partial progress calculations to reward intermediate structural correctness, followed by strict penalties for bad code.

### Base Rewards (Max 1.0)
1. **Correctness (50%)**:
   - Uses robust output parsing (regex extraction and `difflib`) to salvage poorly formatted outputs.
   - For lists: Uses Jaccard Similarity (Intersection over Union).
   - For counts: $Score = 1.0 - \frac{|Diff|}{Truth}$
   - For dicts: Awards partial credit based on correct keys and numerically/textually similar values.
2. **Decomposition Efficiency (30%)**:
   - **Chunking Quality**: Penalizes overlapping lines or lines skipped by the orchestrator.
   - **Overhead Control**: Penalizes spawning too many agents (the "ideal" is ~1 agent per 2000 lines).
   - **Load Balancing**: Encourages even distribution of data across sub-agents.
3. **Prompt Quality (20%)**:
   - A dynamic overlap check measuring how many important keywords from the master task description were successfully relayed into the sub-agent `role_prompt`.

### Strict Penalties (Min -1.0)
To prevent agents from guessing ("random trials") or writing brittle code:
- **Sub-Agent Tracebacks**: `-0.1` per sub-agent that crashes or returns an Error. Tests the orchestrator's ability to write safe Map code.
- **Synthesis Traceback**: `-0.3` if the final synthesis script crashes. Tests the orchestrator's ability to write robust Reduce code.
- **Random Trials (0.0 Correctness)**: `-0.2` if the agent fails to achieve any correctness points.
- **Execution Time Penalty**: `-0.01` per second of map-reduce execution time. Punishes highly inefficient `python_code` (e.g., recursive or unoptimized loop algorithms).
- **Trajectory Penalty**: A decay of `-0.05` per step is applied to penalize endless looping.

## 🚀 Setup & Usage

### Prerequisites
- Python 3.10+ and `uv`
- Docker (Mandatory for safe execution)

### Running the Server (Docker - RECOMMENDED)
This is the only safe way to run the environment.
```bash
# Build the image
docker build -t orchid_env-env:latest -f server/Dockerfile .

# Run the container
docker run -p 7860:7860 orchid_env-env:latest
```

### Running the Server (Local - UNRESTRICTED)
> [!CAUTION]
> Use this only for local development/debugging and only with trusted models.
```bash
uv run python -m server.app
```

### Gradio Web UI
When running the server with `ENABLE_WEB_INTERFACE=true`, a Gradio UI is available at `http://localhost:7860/web`.
